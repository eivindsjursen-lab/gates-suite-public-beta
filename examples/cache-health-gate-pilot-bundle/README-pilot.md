# Cache Health Gate (Private Alpha Pilot)

This bundle contains a local GitHub Action for evaluating cache-regression warnings in CI pull requests.

What it checks:

- cache hit-rate drops
- cache key churn (for example accidental `${{ github.sha }}` in cache keys)
- restore-time regressions (can require tuning in noisy/matrix-heavy repos)

This pilot is intended to be low-risk:

- run in `mode=warn` during alpha
- no third-party SaaS account required
- uses GitHub API via `GITHUB_TOKEN`
- easy rollback by deleting the alpha workflow and local action folder

## Install (assisted pilot)

1. Copy this bundle into your repo at:
   - `.github/actions/cache-health-gate/`
2. Add a dedicated alpha workflow:
   - start from `WORKFLOW-node.yml` or `WORKFLOW-python.yml`
   - keep it separate from your main CI on first pass
3. Run once on default branch:
   - expected onboarding result: `WARN_NO_BASELINE` (or `SKIP_NO_BASELINE` if configured)
4. Build baseline:
   - run `push` on default branch **5-10 times** (profile-dependent)
5. Create one controlled bad PR:
   - inject `${{ github.sha }}` into the cache key to force churn
6. Review:
   - `RESULT`
   - `CONFIDENCE`
   - `BASELINE_SAMPLES`
   - `REASON_CODES`
   - PR comment clarity/actionability

## What to trust first

In private alpha so far, the strongest first signal has been:

- `WARN_HIT_RATE_DROP`

Restore-time warnings (`WARN_RESTORE_REGRESSION`) can appear in good runs in some repos (especially matrix or package-heavy workflows). Treat those as tuning guidance first.

## Expected onboarding behavior

On a fresh repo/workflow, you will often see:

- `WARN_NO_BASELINE` (default behavior), or
- `SKIP_NO_BASELINE` (if configured)

This is expected until enough successful `push` runs on the default branch exist.
